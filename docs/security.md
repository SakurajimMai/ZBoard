# 安全设计文档

## 安全目标

- 商业数据不可被未授权访问。
- 支付回调不可被伪造或重复消费。
- Agent 请求不可被伪造、篡改或重放。
- 订阅 token 泄漏后可以快速重置。
- 后台敏感操作可审计、可追责。
- 数据库、Redis 和节点管理能力不暴露公网。

## 后台安全

必须实现：

- 管理员登录。
- 登录失败限制。
- 管理员状态控制。
- 敏感操作写入审计日志。
- 管理员权限分级。

后续增强：

- 管理员 2FA。
- IP 白名单。
- 敏感操作二次确认。
- 会话设备管理。

## 用户账号安全

- 密码使用强哈希算法保存，例如 Argon2id 或 bcrypt。
- 登录失败需要限流。
- 注册、登录、重置 token 等接口需要防爆破。
- 用户禁用后订阅接口立即失效。

## API 安全

- 用户端 API 使用用户会话或 JWT。
- 管理端 API 使用管理员会话或短期 access token。
- 写接口需要 CSRF 或同等防护，具体取决于认证形态。
- 所有输入必须做 schema 校验。
- 所有列表接口必须分页。
- 所有敏感字段不能直接返回。

## 订阅 token 安全

- token 使用高强度随机值。
- 数据库只保存 token hash。
- 用户可主动重置 token。
- 后台可禁用 token。
- 订阅访问限流。
- 记录订阅访问 IP 和 User-Agent。
- 异常访问可触发提醒或自动禁用。

## Agent 通信安全

Agent 请求必须包含：

```http
X-Zboard-Node-Id
X-Zboard-Timestamp
X-Zboard-Nonce
X-Zboard-Body-SHA256
X-Zboard-Signature
```

校验要求：

- 所有请求使用 HTTPS。
- HMAC-SHA256 签名。
- timestamp 防重放。
- nonce 防重放。
- body hash 防篡改。
- node_secret 只展示一次，服务端只保存 hash。
- node_secret 支持轮换。

## 支付安全

支付回调处理顺序：

1. 保存原始回调。
2. 验证签名。
3. 校验支付渠道事件 ID 幂等。
4. 查询订单。
5. 校验金额。
6. 校验币种。
7. 校验订单状态。
8. 更新支付记录。
9. 更新订单状态。
10. 开通套餐。
11. 生成订阅和节点配置任务。

任何一步失败都必须记录原因，不允许静默丢弃。

## 节点安全

- Agent 不暴露公网管理端口。
- 中央系统不主动 SSH 到节点。
- Agent 使用最小权限运行。
- 运行时配置文件权限隔离。
- 日志脱敏，不输出 node_secret、订阅 token、支付密钥。
- 节点被禁用后不再下发用户配置。

## 基础设施安全

- PostgreSQL 不暴露公网。
- Redis 不暴露公网。
- 管理后台建议通过 HTTPS 访问。
- 当前阶段不配置反向代理；请求体大小限制后续在 API Server 或边界网关中补充。
- 容器间通信使用内部网络。
- 生产环境关闭调试接口。
- `.env` 不提交仓库。

## 审计日志

必须审计：

- 管理员登录。
- 管理员修改用户。
- 管理员修改套餐。
- 管理员修改节点。
- 管理员重新同步配置。
- 管理员回滚配置。
- 管理员禁用用户或节点。
- node_secret 轮换。
- 系统配置修改。

审计内容：

- 操作人类型。
- 操作人 ID。
- 动作。
- 资源类型。
- 资源 ID。
- IP。
- User-Agent。
- 操作详情。
- 创建时间。

## 密钥管理

需要进入环境变量或密钥系统：

- 数据库密码。
- Redis 密码。
- JWT / Session secret。
- 支付渠道密钥。
- Agent 签名服务端配置。
- SMTP 密钥。

不得进入日志：

- 用户密码。
- 订阅 token 明文。
- node_secret 明文。
- 支付私钥。
- 管理员 2FA secret。

## MVP 安全基线

第一版至少完成：

- 密码安全哈希。
- 用户和管理员鉴权。
- 管理后台敏感操作审计。
- 支付回调验签和幂等。
- Agent HMAC 签名。
- 订阅 token hash 存储。
- API 限流。
- PostgreSQL / Redis 内网访问。
- Docker `.env` 与 `.env.example` 分离。
